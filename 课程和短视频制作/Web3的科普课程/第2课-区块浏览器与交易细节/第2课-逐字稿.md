【P2 什么是区块浏览器｜6分钟】
逐字稿（口播）：区块浏览器是什么?简单来说,它是区块链的"搜索引擎"和"账单查询系统"的结合体。

在传统的银行系统中,你要查账单,需要登录网银,输入密码,只能看到自己的记录。但在区块链上,所有的交易都是公开的。任何人都可以通过区块浏览器,查看任何地址的余额、任何交易的详情、任何合约的代码。这就是"公开账簿"的力量。

最常用的区块浏览器包括:
- Etherscan: 以太坊主网及其测试网的首选
- BaseScan: Base链的官方浏览器
- Polygonscan: Polygon链的浏览器
- Arbiscan: Arbitrum的浏览器
- 还有很多其他的,基本上每条主要的链都有自己的浏览器

这些浏览器的界面和功能基本相似,学会了一个,其他的也就会用了。今天我们主要使用Etherscan的Sepolia测试网版本作为示范。

区块浏览器能做什么?

1. 查询交易详情:
输入交易哈希(Transaction Hash),你可以看到这笔交易的所有信息:发送者、接收者、金额、Gas费用、状态(成功或失败)、时间戳、区块高度、调用的合约函数、事件日志等等。

2. 查询地址信息:
输入一个地址,你可以看到:
- 该地址的ETH余额
- 所有的交易历史
- 持有的代币(ERC-20)和NFT(ERC-721)
- 该地址的授权记录
- 如果是合约地址,还能看到合约的源代码

3. 查询区块信息:
输入区块号,你可以看到:
- 这个区块包含的所有交易
- 区块的矿工/验证者
- Gas使用情况
- 区块奖励

4. 查询代币/合约:
输入合约地址,你可以看到:
- 合约的源代码(如果已验证)
- 合约的持有者分布
- 合约的交易历史
- 合约的创建者和创建时间

为什么区块浏览器如此重要?

因为在Web3中,你不能盲目信任任何人的说法。项目方说"交易已完成",你要自己去浏览器验证;钱包显示"成功",你要去浏览器确认状态;有人说"这是官方合约",你要去浏览器查看代码和创建者。

区块浏览器是你的"真相来源"(Source of Truth)。链上记录了什么,就是什么,没有人可以篡改,也没有人可以抵赖。

接下来,我们就要深入到一笔交易的每一个细节中,看看如何从区块浏览器中读取关键信息。

【P3 交易哈希与基础信息｜7分钟】
逐字稿（口播）：当你在钱包里发送一笔交易,或者别人声称给你转了钱,第一件事就是:获取交易哈希(Transaction Hash,也叫TxHash)。

交易哈希是什么?它是这笔交易的唯一标识符,就像快递单号一样。它是一串64个字符的十六进制数,以"0x"开头。例如:
```
0x1234567890abcdef1234567890abcdef1234567890abcdef1234567890abcdef
```

有了这个哈希,你就可以在区块浏览器中查到这笔交易的所有信息。

现在让我们打开Etherscan,以Sepolia测试网为例。访问 https://sepolia.etherscan.io

在首页的搜索框中,粘贴交易哈希,按回车。浏览器会跳转到交易详情页面。

第一眼看到的基础信息包括:

1. Transaction Hash(交易哈希):
这笔交易的唯一ID,可以用来分享、存档、作为凭证。

2. Status(状态):
- Success(成功): 绿色勾,表示交易已上链且执行成功
- Failed(失败): 红色叉,表示交易上链了但执行失败(状态回滚,Gas照扣)
- Pending(待处理): 黄色时钟,表示交易还在内存池中,等待打包

这是最重要的信息。如果不是Success,钱没到账是正常的。

3. Block(区块高度):
这笔交易被打包进哪个区块。点击区块号,可以看到这个区块的所有交易。
区块号越高,表示越新;区块号相同的交易,是同时被打包的。

4. Timestamp(时间戳):
交易被打包的准确时间。注意:这不是你发送交易的时间,而是矿工/验证者打包的时间。如果网络拥堵,这两者可能相差很久。

5. From(发送者):
交易的发起地址。这是用私钥签名这笔交易的账户。

6. To(接收者):
这里要小心。如果是普通转账,To就是收款地址;但如果是合约调用,To是合约地址,而不是最终的资金接收者。

7. Value(ETH金额):
这笔交易直接转移的ETH数量。注意单位:通常显示为ETH,但底层是Wei(1 ETH = 10^18 Wei)。

关键点:Value为0不代表没有转账!

如果你转的是ERC-20代币(如USDC),Value会显示为0 ETH,因为代币的转移是通过合约调用完成的,记录在data字段和事件日志中。

这是新手最容易被误导的地方。你看到Value=0,以为安全,实际上可能正在授权或转移大量代币。

8. Transaction Fee(交易费用):
实际支付的Gas费,等于 Gas Used × Effective Gas Price。
这笔费用已从发送者账户扣除,不可退回。

现在你已经掌握了交易页面的"第一屏"信息。但这只是表面,真正的细节藏在下面。接下来,我们要深入到交易的技术字段中。

【P4 深入交易字段:from/to/value/data｜8分钟】
逐字稿（口播）：现在我们来看交易的核心字段,这些字段决定了"钱往哪里走,权限如何变"。

在交易详情页面往下滚动,你会看到"More Details"或展开的详细信息,包括:

1. Nonce(交易序号):
这是发送者账户的交易计数器,从0开始,每发送一笔交易加1。
Nonce的作用:
- 保证交易顺序:nonce=10的交易必须在nonce=9之后执行
- 防止重放攻击:同一个nonce不能被重复使用

实际问题:如果你的nonce=5的交易卡住了(Gas费太低),后续的nonce=6,7,8...都会排队等待,无法执行。这时候你需要"替换"或"取消"nonce=5的交易,用更高的Gas费重新发送同一个nonce。

2. Position In Block:
这笔交易在区块中的位置。位置靠前的交易先执行,这在DeFi套利、NFT抢购等场景中非常重要。
支付更高的Priority Fee(优先费),可以提高你的交易在区块中的位置。

3. Gas Limit:
你愿意为这笔交易支付的最大Gas数量。这是你设置的上限,不是实际消耗。
如果实际执行需要的Gas超过了Gas Limit,交易会失败,显示"Out of Gas"错误,但已消耗的Gas照样扣除。

4. Gas Used:
这笔交易实际消耗的Gas数量。
关键点:
- Gas Used ≤ Gas Limit
- 如果 Gas Used = Gas Limit,很可能是Gas不够导致失败
- 简单转账(EOA to EOA)通常消耗21,000 Gas
- 合约调用的Gas消耗差异很大,从几万到几百万不等

5. Gas Price / Base Fee + Priority Fee:
在EIP-1559之后,Gas费用分为两部分:
- Base Fee: 网络动态调整的基础费用,会被销毁
- Priority Fee (Max Priority Fee): 你愿意额外支付给验证者的小费

Total Gas Fee = Gas Used × (Base Fee + Priority Fee)

你在钱包里看到的"慢/中/快"选项,本质上是在调整Priority Fee的大小。

6. Input Data(也叫Calldata):
这是最复杂也是最关键的字段。

如果是简单的ETH转账(EOA to EOA),Input Data会显示为"0x",表示没有附加数据。

但如果是合约调用,Input Data会是一长串十六进制数据,格式如下:
- 前4个字节(8个十六进制字符)是函数选择器(Function Selector),用来指定调用哪个函数
- 后面是函数参数,经过ABI编码

举个例子,如果你调用ERC-20代币的transfer函数,Input Data可能是:
```
0xa9059cbb
0000000000000000000000001234567890abcdef1234567890abcdef12345678
0000000000000000000000000000000000000000000000000de0b6b3a7640000
```

解读:
- `0xa9059cbb`: transfer(address,uint256)的函数选择器
- 第二行: 接收地址(去掉前导0)
- 第三行: 转账金额,这里是0de0b6b3a7640000(十六进制)= 1,000,000,000,000,000,000 = 1 × 10^18 = 1个代币(如果精度是18位)

好消息是,Etherscan会帮你解析Input Data。在"Input Data"字段下,你会看到"Decode Input Data"按钮,点击后可以看到人类可读的格式:
```
Function: transfer(address _to, uint256 _value)
MethodID: 0xa9059cbb
[0]: 0x1234567890abcdef1234567890abcdef12345678
[1]: 1000000000000000000
```

这样你就能清楚地知道:这笔交易调用了transfer函数,向地址0x1234...转了1个代币。

关键安全提示:
- 每次签名前,务必理解你在调用什么函数
- 如果Input Data很长,而你又不确定它的作用,不要盲目签名
- 特别警惕approve、setApprovalForAll等授权函数,以及transferFrom等可以划走你资产的函数

记住:from/to/value/data这四个字段,完整描述了"谁、对谁、转了多少、做了什么"。看懂这四个字段,你就掌握了交易的本质。

【P5 状态改变:Internal Transactions｜8分钟】
逐字稿（口播）：现在我们来讲一个容易被忽略的重要概念:内部交易(Internal Transactions,也叫Internal Txns)。

什么是内部交易?

当你调用一个智能合约,这个合约可能会再去调用其他合约,或者给其他地址转账。这些"合约发起的子操作",就叫内部交易。

注意:内部交易不是独立的交易,不需要单独签名,也不直接消耗额外的Gas(已包含在主交易的Gas中)。它们是主交易执行过程中的"附带效应"。

举个现实例子:

你在Uniswap上用USDC换ETH。你发起的主交易是:
- From: 你的地址
- To: Uniswap Router合约
- Value: 0
- Data: swapExactTokensForETH(...)

在这笔交易执行过程中,会发生一系列内部操作:
1. Uniswap Router调用USDC合约,从你的地址划走USDC(这需要你之前已经授权)
2. Uniswap Router调用流动性池合约,执行兑换逻辑
3. 流动性池合约计算兑换比例,更新池子状态
4. 流动性池合约将ETH转给Uniswap Router
5. Uniswap Router将ETH转给你

步骤1、2、3、4、5都是内部交易。你在主交易的详情页面中,只能看到From(你)、To(Uniswap Router)、Value(0),看不到真正的资金流动细节。

那么,如何查看内部交易?

在Etherscan的交易详情页面,点击"Internal Txns"标签。这里会列出所有的内部转账和合约调用。

示例截图说明:

假设交易哈希是0xabc123...,点击Internal Txns,你可能看到:
```
From: Uniswap Router (0xdef456...)
To: 你的地址 (0x789abc...)
Value: 0.5 ETH
Type: call
```

这告诉你:虽然主交易的Value是0,但通过内部交易,你实际收到了0.5 ETH。

为什么内部交易很重要?

1. 真实的资金流向:
主交易的Value字段只显示你直接发送的ETH,但通过合约调用,你可能收到或失去更多资产。内部交易记录了这些"隐藏"的转移。

2. 复杂DeFi操作的透明化:
DeFi协议通常涉及多个合约的组合调用。通过内部交易,你可以追踪每一步的资金流动,理解协议的实际运作方式。

3. 安全审计:
如果一个合约声称"只收取1%手续费",你可以通过内部交易验证:它是否真的只拿了1%?有没有偷偷转给其他地址?

4. 调试失败交易:
有时候主交易显示失败,但不清楚是哪一步出错。通过内部交易,你可以定位:是代币转账失败?还是池子流动性不足?还是滑点保护触发?

真实案例:

2023年初,某个DeFi协议声称"零手续费"。但用户通过查看内部交易发现,协议在兑换过程中,会向一个未公开的地址转移0.3%的代币,这实际上是隐藏的手续费。社区质疑后,项目方才承认并修改了文案。

如何理解Internal Txns列表:

通常会显示:
- Type: call(普通调用)、delegatecall(委托调用,使用调用者的状态)、create(创建新合约)
- From/To: 调用的发起者和接收者
- Value: 转移的ETH数量
- Gas Limit / Gas Used: 这一步的Gas消耗

注意:delegatecall比较特殊,它会在调用者的上下文中执行被调用合约的代码。这是代理合约(Proxy)实现可升级性的关键机制,但也是安全风险点:如果delegatecall的目标是恶意合约,它可以直接修改调用者的状态。

实践建议:

每次执行复杂的DeFi操作(如swap、借贷、质押)后,养成查看Internal Txns的习惯:
- 资金最终去了哪里?
- 有没有预期之外的转账?
- 手续费是否合理?

通过内部交易,你可以"看穿"合约的黑盒,理解真实的资金流向。这是高阶用户和审计人员的必备技能。

【P6 事件日志:Event Logs详解｜7分钟】
逐字稿（口播）：上节课我们讲过,事件(Event Logs)不是状态,而是合约执行过程中主动发出的"操作日志"。现在我们来学习如何在区块浏览器中读取和理解事件日志。

在交易详情页面,点击"Logs"标签,你会看到这笔交易发出的所有事件。

事件的结构:

每个事件包含:
1. Address: 发出事件的合约地址
2. Name: 事件的名称(如果合约已验证)
3. Topics: 事件的索引参数
4. Data: 事件的非索引参数

举个具体的例子:ERC-20代币的Transfer事件。

当你转账100个USDC给Bob,USDC合约会发出一个Transfer事件:
```
event Transfer(address indexed from, address indexed to, uint256 value);
```

在Etherscan的Logs中,你会看到:
```
Address: 0xA0b8...5e (USDC合约地址)
Name: Transfer (address from, address to, uint256 value)
Topics:
  [0]: 0xddf252ad1be2c89b69c2b068fc378daa952ba7f163c4a11628f55a4df523b3ef (事件签名的哈希)
  [1]: 0x000000000000000000000000<你的地址>
  [2]: 0x000000000000000000000000<Bob的地址>
Data:
  value: 100000000 (100 USDC,精度6位)
```

解读:
- Topics[0]: 是事件签名`Transfer(address,address,uint256)`的Keccak256哈希,用来快速识别事件类型
- Topics[1]: from地址(索引参数)
- Topics[2]: to地址(索引参数)
- Data: 转账金额(非索引参数)

为什么要区分索引(indexed)和非索引参数?

索引参数可以被快速检索。比如你想查询"所有转给Bob的交易",可以通过Topics[2]筛选,效率很高。非索引参数存储在Data中,不能直接用于检索,但可以存储更复杂的数据。

Solidity规定:一个事件最多有3个indexed参数(加上Topics[0]的事件签名,总共4个Topics)。

常见的ERC-20事件:

1. Transfer(address indexed from, address indexed to, uint256 value):
代币转账,记录从谁、到谁、转了多少。

2. Approval(address indexed owner, address indexed spender, uint256 value):
授权事件,记录owner授权spender可以操作多少代币。

查看这个事件,你可以知道:你授权了哪个合约、授权了多少额度。定期检查Approval事件,可以发现是否有未预期的授权。

常见的ERC-721(NFT)事件:

1. Transfer(address indexed from, address indexed to, uint256 indexed tokenId):
NFT转移,记录编号tokenId的NFT从from转给to。

2. Approval(address indexed owner, address indexed approved, uint256 indexed tokenId):
单个NFT的授权。

3. ApprovalForAll(address indexed owner, address indexed operator, bool approved):
授权某个地址操作你所有的NFT。这是高风险操作,务必谨慎。

如何利用事件日志:

1. 验证操作结果:
钱包显示"转账成功",你可以通过查看Transfer事件,确认金额和接收者是否正确。

2. 追踪授权历史:
查看Approval和ApprovalForAll事件,了解你给了哪些合约什么权限。

3. 分析合约行为:
通过事件日志,可以推断合约内部的逻辑。比如一个DeFi协议,你可以看它发出了哪些事件,涉及哪些地址,从而理解它的运作机制。

4. 监控异常活动:
如果你没有发起任何操作,但看到你的地址出现在Transfer或Approval事件中,说明可能:
- 有人给你转账(好事)
- 你之前授权的合约在执行操作(检查是否合理)
- 你的私钥泄露,有人在盗取资产(立即转移资产)

事件 vs 状态的再次强调:

事件是"操作记录",方便查询和展示,但不是权威来源。

真实案例:某个恶意合约,故意发出虚假的Transfer事件,显示"你收到了1000个某代币"。一些不成熟的钱包或DApp只监听事件,就会在界面上显示你有1000个代币。但实际上,合约的余额状态根本没变,你的余额仍然是0。

所以,关键数据(如余额、授权额度)一定要回读合约状态,不能只信事件。

实践建议:

每次执行重要操作后,养成查看Logs的习惯:
- 发出了哪些事件?
- 事件的参数是否符合预期?
- 有没有预期之外的事件?

通过事件日志,你可以快速理解交易做了什么,验证是否符合预期,发现潜在的问题。

【P7 Gas费用详解:估算与优化｜8分钟】
逐字稿（口播）：Gas费用是Web3中每个人都关心的话题。今天我们要深入理解Gas的计算机制,以及如何合理估算和优化费用。

在交易详情页面,Gas相关的字段包括:

1. Gas Limit:
你设置的最大Gas消耗量。这是你的"预算上限"。

2. Gas Used by Transaction:
实际消耗的Gas数量。通常会显示使用比例,比如"21,000 (100%)"或"150,000 (75%)"。

如果使用比例是100%,即Gas Used = Gas Limit,很可能是Gas不够导致失败。

3. Gas Price:
在EIP-1559之前的简单模式下,Gas Price是你愿意支付的每单位Gas的价格,单位是Gwei(1 Gwei = 10^-9 ETH)。

Total Fee = Gas Used × Gas Price

4. Base Fee + Max Fee + Max Priority Fee:
在EIP-1559之后:
- Base Fee: 网络动态调整的基础费用,每个区块都不同,根据前一个区块的使用率自动调整
- Max Fee Per Gas: 你愿意支付的最高总费用(Base + Priority)
- Max Priority Fee Per Gas: 你愿意给验证者的最高小费

实际支付的费用:
```
Effective Gas Price = min(Max Fee, Base Fee + Max Priority Fee)
Total Fee = Gas Used × Effective Gas Price
```

Etherscan会显示:
- Base Fee: 30 Gwei (由网络决定,你无法控制)
- Max Fee: 50 Gwei (你设置的上限)
- Max Priority Fee: 2 Gwei (你愿意额外支付的小费)
- Effective Gas Price: 32 Gwei (实际支付 = Base 30 + Priority 2,低于Max 50)

5. Transaction Fee:
最终支付的总费用,单位ETH和美元(基于当时的ETH价格)。

举例:
```
Gas Used: 100,000
Base Fee: 30 Gwei
Max Priority Fee: 2 Gwei
Total Fee = 100,000 × (30 + 2) = 3,200,000 Gwei = 0.0032 ETH ≈ $6.40 (假设ETH = $2000)
```

其中:
- Base Fee部分(30 × 100,000 = 3,000,000 Gwei)被销毁,不归验证者
- Priority Fee部分(2 × 100,000 = 200,000 Gwei)归验证者,作为打包激励

如何估算合理的Gas费用?

1. 参考历史数据:
Etherscan首页有"Gas Tracker",显示当前的:
- Low(慢): 适合不着急的交易,可能需要等几分钟到十几分钟
- Average(中): 通常1-2分钟内打包
- High(快): 几乎立即打包,适合紧急情况

你可以根据场景选择对应的Priority Fee。

2. 使用钱包的估算功能:
MetaMask等钱包会根据网络状况,自动建议Gas Limit和Fee。通常是比较准确的,但也可能偏高(为了确保成功)。

3. 模拟交易:
一些工具(如Tenderly、Blocknative)可以模拟交易,预测Gas消耗和费用,帮助你在发送前做决策。

4. 观察网络拥堵:
- 工作日白天(美国/欧洲时间)通常较贵
- 周末和深夜(UTC)通常较便宜
- 有热门NFT发售、DeFi事件、市场剧烈波动时,Gas会飙升

常见的Gas优化策略:

1. 批量操作:
如果要做多笔转账,尽量合并成一笔交易,减少固定成本(每笔至少21,000 Gas)。

2. 使用Layer 2:
Arbitrum、Optimism等L2的Gas费只有主网的几十分之一。对于高频小额操作,L2是更好的选择。

3. 选择低峰时段:
非紧急操作,等到Gas便宜时再做。可以设置Gas Price Alert,当费用低于某个阈值时通知你。

4. 避免复杂合约:
同样的功能,有些合约写得更高效,消耗更少Gas。选择经过优化的协议。

5. 取消无效交易:
如果交易卡在内存池里很久,与其等待,不如用更高Gas替换或取消,避免后续交易都被卡住。

真实案例:Gas War

2021年5月,Bored Ape Yacht Club的后续项目Otherside土地销售。由于设计不当,没有使用荷兰拍卖或白名单,而是"先到先得",导致数万人同时发送交易抢购。

Gas费瞬间飙升到8000+ Gwei,很多人设置Max Fee为几千美元,最终:
- 成功铸造的人,单笔交易花费了数千美元Gas
- 失败的人,同样损失了数千美元Gas
- 整个事件烧掉了超过1.7亿美元的ETH(其中大部分被销毁)

教训:
- 不要盲目参与Gas War,冷静评估成本
- 项目方应该设计更合理的机制,避免Gas竞价

如何在Etherscan查看交易的Gas效率?

1. 查看Gas Used占Gas Limit的比例:
- 接近100%: 可能Gas不够,或者刚好够(高风险)
- 50-80%: 合理范围
- 很低(如10%): 可能Gas Limit设置过高,浪费(但不影响实际费用,因为只扣Used部分)

2. 对比类似交易:
在合约页面,查看其他人执行同样函数的Gas消耗,可以作为参考。

3. 查看交易的"Gas Used by Txn":
如果远高于预期,可能是合约写得不够高效,或者触发了意外的逻辑分支。

记住:Gas费是不可退的,即使交易失败。所以,发送前务必:
- 确认余额够(ETH余额 ≥ Value + Gas Fee)
- 确认授权足(如果是代币操作)
- 合理设置Gas Limit,不要太低也不要太高
- 根据紧急程度选择合适的Priority Fee

【P8 交易状态:Success vs Failed｜7分钟】
逐字稿（口播）：交易上链了,但这不代表它成功了。今天我们要学会区分Success和Failed,以及如何定位失败原因。

在Etherscan交易详情页面,第一个字段就是Status:

1. Success (绿色勾):
交易已被打包进区块,并且成功执行,状态已改变。
- 如果是转账,钱已到账
- 如果是授权,权限已设置
- 如果是合约调用,逻辑已执行

2. Failed (红色叉):
交易已被打包进区块,但执行失败,状态回滚到执行前。
- 如果是转账,钱没有转出
- 如果是授权,权限没有设置
- 但是,Gas费照样扣除,因为验证者已经做了计算

3. Pending (黄色时钟):
交易还在内存池(Mempool)中等待打包,尚未上链。

为什么会失败?

Etherscan会在Status旁边显示失败原因(如果有的话)。常见的失败类型:

1. Out of Gas:
Gas Limit不够,交易执行到一半,Gas用完了。

示例:
```
Status: Fail with error 'Out of Gas'
Gas Limit: 50,000
Gas Used: 50,000 (100%)
```

解决方法:
- 提高Gas Limit。你可以看类似交易的Gas Used,设置略高一些
- 注意:Gas Limit过高不会多扣费,只是上限;但如果真的消耗了,费用会很高

2. Revert:
合约主动拒绝执行,并可能返回错误信息。

示例:
```
Status: Fail with error 'ERC20: transfer amount exceeds balance'
```

这表示:你试图转账的金额超过了你的余额。

常见的Revert原因:

a) 余额不足:
```
ERC20: transfer amount exceeds balance
```
你的代币余额不够。

b) 未授权:
```
ERC20: transfer amount exceeds allowance
```
你没有授权合约操作你的代币,或者授权额度不够。

解决:先调用approve函数,授权足够的额度。

c) 滑点保护触发:
```
UniswapV2Router: INSUFFICIENT_OUTPUT_AMOUNT
```
在Uniswap等DEX上交易时,价格变化超过了你设置的滑点容忍度。

解决:放宽滑点设置,或者等市场稳定后再操作。

d) 逻辑条件不满足:
```
Not in whitelist
Already claimed
Exceeds max supply
```
合约设置了某些限制条件,你不满足。

解决:这是合约设计的限制,通常无法绕过,除非你满足条件。

e) 权限不足:
```
Ownable: caller is not the owner
```
这个函数只能由合约的owner调用,你没有权限。

f) 重入攻击保护:
```
ReentrancyGuard: reentrant call
```
合约检测到重入调用,拒绝执行,这是安全机制。

g) 自定义错误:
有些合约会定义自己的错误信息,需要查看合约代码才能理解。

如何查看详细的错误信息?

1. 在Etherscan的Status字段,点击"Click to see More"或错误信息本身,会展开详细的revert reason。

2. 如果合约已验证,Etherscan会尝试解析错误信息,显示人类可读的文本。

3. 如果没有明确的错误信息,只显示"Fail",你需要:
- 查看Event Logs,看执行到哪一步
- 查看Internal Txns,看哪个子调用失败了
- 使用Tenderly等调试工具,模拟交易,逐步追踪

真实案例:

用户A想在Uniswap上用1000 USDC换ETH。他设置了0.5%的滑点保护。交易发送后,由于网络延迟,在打包前已经有大额交易改变了池子的价格。等A的交易执行时,价格变化超过0.5%,Uniswap合约检查不通过,Revert并返回"INSUFFICIENT_OUTPUT_AMOUNT"。

交易失败,A的USDC没动,但Gas费(假设$5)被扣除。

A的选择:
- 放宽滑点到1%,重新发送交易
- 等待市场稳定,再操作
- 分批交易,减少单笔影响

如何避免失败?

1. 发送前检查:
- 余额够不够?(ETH和代币)
- 授权了吗?额度够吗?
- Gas Limit合理吗?(可以参考类似交易)
- 滑点设置合理吗?

2. 使用模拟工具:
很多钱包(如MetaMask)在你点击确认前,会模拟交易,预测是否会失败。如果模拟失败,会警告你。

3. 小额测试:
对于复杂操作,先用小额测试,确认流程没问题,再执行大额。

4. 查看合约逻辑:
如果不确定合约会做什么,先在Etherscan查看代码,或者查社区讨论,了解常见的坑。

记住:失败的交易,Gas费不退。所以,提前检查,避免不必要的损失。一笔$5的失败Gas费看起来不多,但如果频繁失败,累积起来就是可观的浪费。

【P9 区块确认与最终性｜6分钟】
逐字稿(口播):交易显示Success了,是不是就绝对安全了?不一定。今天我们要理解"确认"和"最终性"的概念。

在Etherscan的交易详情页面,你会看到:

Block: 12345678
Block Confirmations: 15 block confirmations

什么是确认(Confirmations)?

确认数是指:在你的交易所在区块之后,又产生了多少个新区块。

计算方式:
```
Confirmations = Current Block Height - Transaction Block Height
```

举例:
- 你的交易在区块12345678
- 当前最新区块是12345693
- Confirmations = 12345693 - 12345678 = 15

为什么需要多个确认?

在区块链中,理论上存在"区块重组"(Reorg)的可能:如果网络发生短暂分叉,你的交易所在的链可能被"孤立"(orphaned),交易需要重新打包。

确认数越多,你的区块被重组的概率越低,交易越"不可逆"。

不同场景的确认要求:

1. 小额支付(如咖啡):
1-2个确认就够,对应10-30秒(以太坊PoS下,每个区块约12秒)。
风险:几乎可以忽略,除非遭遇51%攻击(在以太坊上几乎不可能)。

2. 中等金额(如几百美元):
5-10个确认,对应1-2分钟。
这是大部分DApp的默认要求。

3. 大额交易(如数万美元):
等待"最终性"(Finality),对应2个epoch,约12-15分钟。
交易所在处理充值时,通常要求更多确认,比如20-50个,对应5-10分钟。

4. 跨链桥:
从L2提款到L1,需要等待"挑战期"或"证明生成":
- Optimistic Rollup(如Arbitrum、Optimism): 7天挑战期
- ZK Rollup(如zkSync): 几小时到1天,等待ZK证明生成并在L1验证

以太坊PoS的最终性:

2022年9月,以太坊完成"合并"(The Merge),从PoW切换到PoS。PoS引入了"最终性"(Finality)的概念:

- Epoch: 32个区块为一个epoch,约6.4分钟
- Checkpoint: 每个epoch的第一个区块
- Finalized: 当一个checkpoint获得了2/3以上验证者的投票,它就进入"已最终确定"状态,理论上不可逆

在Etherscan上,你可以看到:
```
Finalized: Yes (after 2 epochs)
```

一旦Finalized,除非发生极端的"深度重组"(需要51%的验证者合谋并接受巨额罚没),否则交易不可能被撤销。

实践建议:

1. 日常小额操作:
1-2个确认就继续,不用等太久。

2. 重要操作(如大额转账、NFT交易):
等5-10个确认,或者等Finalized标记出现。

3. 交易所充值:
交易所会有自己的确认要求,通常是20-50个确认,耐心等待。如果显示"Pending",不要重复充值,等确认够了会自动到账。

4. 跨链操作:
理解目标链的确认机制:
- 侧链(如Polygon): 有自己的共识,几秒到几分钟
- Optimistic Rollup: 提款需要7天挑战期,这是设计特性,无法绕过
- ZK Rollup: 提款需要等ZK证明,通常几小时

如何查看当前网络的确认速度?

Etherscan首页显示:
- Average Block Time: 12秒(以太坊PoS)
- Finality Time: 约12-15分钟(2 epochs)

你可以根据这个估算你的交易何时达到足够的确认。

真实案例:

2017年,某交易所只要求1个确认就认为充值成功。攻击者利用这个漏洞,在网络短暂分叉时,向交易所充值,快速交易提现,然后让充值交易所在的链被孤立,充值"消失"了,但提现已经完成。交易所损失了数十万美元。

此后,交易所普遍提高了确认要求,至少20-50个确认,大大降低了这种攻击的可行性。

记住:
- 确认越多越安全,但也越慢
- 根据场景权衡:小额求快,大额求稳
- 了解不同链的确认机制,设置合理的等待时间

【P10 定位失败原因:实战排错｜8分钟】
逐字稿(口播):现在我们来做一个完整的排错演示,教你如何像侦探一样,从交易详情中定位失败的准确原因。

场景:你在某个DeFi协议上尝试质押代币,交易失败了,钱包显示红色的"Failed",Gas费被扣了,但不知道为什么失败。

排错步骤:

步骤1: 获取交易哈希,打开Etherscan

从钱包的交易历史中,复制交易哈希,粘贴到Etherscan搜索框。

步骤2: 查看Status

Status: Fail

这确认了交易确实失败了。

步骤3: 查看错误信息

在Status旁边,你看到:
```
Fail with error 'ERC20: transfer amount exceeds allowance'
```

这已经很明确了:你的授权额度不够。

步骤4: 验证根因

为什么授权不够?两种可能:
1. 你从未授权过这个合约
2. 你之前授权了,但额度已经用完,或者授权被撤销了

如何验证?

方法A: 查看你的授权历史
- 访问 https://revoke.cash
- 连接钱包,选择对应链
- 查看你对这个协议的授权额度

方法B: 直接查询合约状态
- 在Etherscan上,进入代币合约页面
- 点击"Read Contract"
- 找到`allowance(address owner, address spender)`函数
- 输入你的地址(owner)和协议合约地址(spender)
- 点击Query,查看返回的额度

假设返回0,说明你没有授权,或者授权已经用完。

步骤5: 解决方案

你需要先授权,再重新质押。

1. 调用代币合约的`approve`函数:
- To: 代币合约地址(如USDC)
- Function: approve(address spender, uint256 amount)
- spender: 协议合约地址
- amount: 你要授权的额度(建议根据实际需要,不要无限授权)

2. 等待授权交易确认(Status: Success)

3. 重新发送质押交易

案例2: Out of Gas

场景:你在铸造NFT,交易失败,错误信息是"Out of Gas"。

排错步骤:

步骤1: 查看Gas相关字段
```
Gas Limit: 100,000
Gas Used: 100,000 (100%)
```

Gas Used = Gas Limit,说明Gas不够,执行到一半就用完了。

步骤2: 查看类似交易

在合约页面,点击"Txns"标签,查看其他人铸造同样NFT的交易,看他们的Gas Used是多少。

假设你看到大部分成功的交易,Gas Used在150,000左右。

步骤3: 解决方案

重新发送交易,将Gas Limit提高到200,000(留一些余地),确保够用。

注意:Gas Limit设高不会多扣费,只是上限;实际只扣Gas Used × Gas Price。

案例3: 滑点保护触发

场景:你在Uniswap上兑换代币,交易失败,错误信息是"UniswapV2Router: INSUFFICIENT_OUTPUT_AMOUNT"。

排错步骤:

步骤1: 理解错误含义

这个错误表示:实际能兑换到的代币数量,少于你设置的最低接受量(由滑点保护决定)。

步骤2: 查看Input Data

在交易详情页,点击"Decode Input Data",你可能看到:
```
Function: swapExactTokensForTokens
amountIn: 1000000000 (1000 USDC,精度6位)
amountOutMin: 995000000000000000 (0.995 ETH,精度18位,对应0.5%滑点)
```

步骤3: 分析原因

你设置了0.5%的滑点保护。但在交易打包时,池子价格已经变化,实际能兑换到的ETH少于0.995,所以合约拒绝执行。

步骤4: 解决方案

选项A: 放宽滑点到1%或2%,重新发送交易
选项B: 等待市场稳定,再操作
选项C: 分批交易,减少单笔对价格的影响

案例4: 权限不足

场景:你尝试调用某个函数,失败,错误信息是"Ownable: caller is not the owner"。

分析:
这个函数只能由合约的owner调用,你不是owner,所以被拒绝。这是合约设计的权限控制,你无法绕过。

如何查看合约的owner?
- 在Etherscan合约页面,点击"Read Contract"
- 查找`owner()`或`admin()`函数
- 调用,查看返回的地址

如果你不是owner,就无法调用这类受限函数。

排错通用清单:

遇到失败交易,按这个顺序检查:

1. 看Status错误信息(最直接)
2. 看Gas Used vs Gas Limit(是否Out of Gas)
3. 看Event Logs(执行到哪一步,发出了哪些事件)
4. 看Internal Txns(哪个子调用失败了)
5. 看Input Data(调用了什么函数,参数是否合理)
6. 查询合约状态(余额、授权、权限是否满足)
7. 参考社区讨论(其他人是否遇到类似问题)

工具推荐:

1. Tenderly:
可以"重放"失败的交易,逐步调试,看每一步的状态变化,精确定位问题。

2. Blocknative:
交易模拟工具,在发送前预测是否会失败,以及Gas消耗。

3. Revoke.cash:
查看和管理你的授权,避免授权不足或过度授权。

记住:每一笔失败都是学习的机会。通过排错,你会越来越熟悉合约的逻辑,越来越能预判风险,最终成为一个真正的"链上侦探"。

【P11 区块时间与网络状态｜5分钟】
逐字稿(口播):交易的速度和成本,很大程度上取决于网络的当前状态。今天我们学习如何查看网络状态,做出更明智的决策。

在Etherscan首页,你可以看到实时的网络状态:

1. ETH Price:
当前以太币价格,影响Gas费用的美元成本。

2. Market Cap:
以太坊的市值,反映网络的整体规模。

3. Transactions:
24小时内的总交易数,反映网络活跃度。

4. Med Gas Price (Median Gas Price):
中位数Gas价格,单位Gwei。这是当前大部分交易使用的费用水平。

5. Gas Tracker:
实时的Gas费用建议,通常分为:
- Low (慢): 适合不着急的交易
- Average (中): 通常1-2分钟
- High (快): 几乎立即

点击"Gas Tracker"页面,你可以看到更详细的信息:
- Base Fee: 当前区块的基础费用
- Priority Fee: 建议的小费
- 历史趋势图: 过去24小时/7天的Gas价格变化

如何利用Gas Tracker优化成本?

1. 观察趋势:
如果当前Gas在高位,而趋势图显示通常在几小时后会降低,你可以选择等待。

2. 选择低峰时段:
一般规律:
- 周末比工作日便宜
- 深夜(UTC时区)比白天便宜
- 避开重大事件(NFT发售、DeFi挖矿、市场暴跌等)

3. 设置Gas Price Alert:
一些工具(如GasNow、Blocknative)可以设置警报:当Gas低于某个阈值时通知你。

4. 使用Layer 2:
如果Gas持续高位,考虑将资产桥接到L2(如Arbitrum、Optimism),在那里操作,费用只有主网的几十分之一。

区块时间(Block Time):

以太坊PoS后,区块时间稳定在约12秒。这意味着:
- 1个确认 ≈ 12秒
- 5个确认 ≈ 1分钟
- 最终性(2 epochs) ≈ 12-15分钟

你可以根据这个估算交易何时达到足够的安全性。

网络拥堵的影响:

当网络拥堵时:
- Base Fee会快速上升(每个区块最多调整12.5%)
- 交易在内存池中排队时间变长
- 如果你的Gas Price太低,可能长时间无法打包,甚至被"卡住"

如何判断网络是否拥堵?

1. 查看Pending Transactions:
Etherscan首页显示"Pending Txns"数量。如果数量很高(如数万),说明拥堵。

2. 查看区块的Gas使用率:
在区块详情页面,看"Gas Used / Gas Limit"。如果接近100%,说明区块满载,竞争激烈。

3. 查看Base Fee变化:
如果Base Fee在短时间内快速上升,说明需求激增。

真实案例:

2023年,某个热门NFT项目Mint,几千人同时抢购。网络瞬间拥堵,Pending Txns超过10万,Base Fee从30 Gwei飙升到300+ Gwei,很多人为了抢到,设置了极高的Priority Fee。

最终,成功Mint的人,单笔Gas费高达几百美元;失败的人,也损失了几十到上百美元的Gas。

教训:
- 不要在极端拥堵时盲目参与
- 评估"Gas成本+NFT成本"的总价值,是否合理
- 如果项目方设计合理(如荷兰拍卖、分批Mint),可以避免Gas War

实践建议:

1. 发送交易前,看一眼Gas Tracker,评估当前费用是否可接受。

2. 如果不着急,设置较低的Priority Fee,慢慢等;如果紧急,提高Priority Fee,加速打包。

3. 定期(如每天)查看网络状态,了解什么时段Gas最便宜,形成操作习惯。

4. 对于高频操作(如交易、游戏),优先使用Layer 2,避免被主网Gas"割肉"。

- Gas Used是Limit的80%,说明Gas设置合理,留有余地

通过这笔交易,我们确认了:
1. 交易成功(Status: Success)
2. 调用了transfer函数,转了100代币(Input Data)
3. 事件日志验证了转账(Logs)
4. Gas费用合理(Gas Used < Limit)

【P15 安全实践:识别可疑交易｜7分钟】
逐字稿(口播):掌握了如何查看交易,接下来我们要学习如何识别可疑和恶意的交易,保护自己的资产安全。

可疑信号清单:

1. 未预期的授权(Approval):
如果你在Logs中看到Approval事件,但你没有主动操作,要立即警惕:
- 检查授权给了谁(spender地址)
- 检查授权了多少(value,特别警惕无限授权)
- 如果不认识这个合约,立即撤销授权

真实案例:
用户A点击了一个钓鱼网站的"领取空投"按钮,签署了一个看起来"无害"的交易。事后查看Logs,发现发出了Approval事件,授权了一个陌生合约无限操作他的USDC。几分钟后,该合约调用transferFrom,转走了他所有的USDC。

防范:
- 任何Approval操作,必须明确知道授权给谁
- 定期用Revoke.cash检查授权列表
- 撤销不再使用的授权

2. 未预期的transferFrom:
如果你看到Transfer事件,from是你的地址,但你没有主动发起转账,说明:
- 某个你之前授权的合约,正在操作你的资产
- 检查是否是你预期的操作(如DeFi协议的自动复投)
- 如果不是,你的授权可能被滥用,立即撤销

3. 可疑的合约地址:
如果交易的To地址是合约,你需要:
- 在Etherscan查看合约是否已验证(Verified Contract)
- 如果未验证,极度危险,无法看到代码
- 如果已验证,查看合约代码,关注:
  * 有没有owner或admin,权限是什么
  * 有没有后门函数(如rug pull、暂停、拉黑)
  * 是否可升级(Proxy模式)

真实案例:
某个"DeFi挖矿"项目,合约未验证。用户质押代币后,发现无法提取。查看合约字节码(高难度),发现有一个隐藏函数,只有owner可以调用,直接转走池子里的所有资产。项目方跑路,用户血本无归。

防范:
- 只与已验证的合约交互
- 查看合约的审计报告(如CertiK、PeckShield等)
- 社区评价和历史记录

4. 异常的Gas消耗:
如果一个简单的操作(如转账),Gas Used异常高,可能:
- 合约写得低效
- 触发了复杂的逻辑(如循环、递归)
- 合约有Bug或恶意代码

对比类似交易的Gas消耗,如果明显偏高,要警惕。

5. 频繁的失败交易:
如果你看到某个地址或合约,有大量失败交易,可能:
- 合约有Bug
- 用户不熟悉操作,频繁失败
- 攻击者在尝试exploit

避免与这类合约交互。

6. Internal Txns中的意外转账:
如果内部交易中,有资金转向未知地址,要警惕:
- 是否是隐藏的手续费?
- 是否是项目方的后门?

真实案例:
某个"零手续费"DEX,通过Internal Txns,每次swap都向项目方地址转0.3%的代币,用户不查Internal Txns根本不知道。

7. 可疑的Event Logs:
有些恶意合约,会故意发出虚假事件,混淆用户:
- 比如发出Transfer事件,显示"你收到了100个某代币"
- 但实际状态没变,余额仍然是0

验证方法:
- 不要只看事件,要查询合约状态(balanceOf)
- 用钱包刷新余额,看是否真的增加

识别工具:

1. MetaMask等钱包的交易模拟:
在你签名前,会模拟交易,预测结果。如果显示"可疑活动"或"高风险",不要签。

2. 安全插件:
- Pocket Universe
- Fire
- Wallet Guard

这些插件会在你签名前,分析交易的风险,提示可疑的授权、转账、合约调用。

3. Etherscan的Token Approval页面:
在你的地址页面,点击"Token Approvals",查看所有授权,及时撤销。

实践建议:

1. 每次签名前,问自己:
- 我在调用什么函数?
- 会改变什么状态?
- 授权给谁?额度多少?
- 这个合约可信吗?

2. 签名后,立即查看交易详情:
- Status是否成功?
- Logs发出了哪些事件?
- Internal Txns有没有意外转账?

3. 定期审查:
- 每月用Revoke.cash检查授权
- 查看地址的交易历史,发现异常及时处理

4. 小额测试:
- 对于新协议,先用小额测试,确认安全后再投入大额

记住:在Web3中,你是自己的安全负责人。多一分警惕,少一分损失。

【P16 课程总结与关键点回顾｜5分钟】
逐字稿(口播):好,我们来总结今天学到的核心内容。

今天这节课,我们深入学习了如何使用区块浏览器,看懂交易的每一个细节,定位失败原因,优化Gas费用,识别可疑交易。

核心知识点回顾:

1. 区块浏览器的作用:
链上的"真相来源",用来验证、追踪、审计所有交易和合约。

2. 交易的核心字段:
- Hash: 唯一标识
- Status: Success / Failed / Pending
- From / To / Value / Data: 谁、对谁、转了多少、做了什么
- Nonce: 交易序号,保证顺序
- Gas Limit / Used / Price: 费用机制

3. Input Data的重要性:
不要只看Value,要看Data里调用了什么函数,传了什么参数。Value=0不代表安全。

4. Internal Transactions:
合约内部的子调用和转账,揭示真实的资金流向。

5. Event Logs:
操作日志,方便查询,但不是状态,要以状态为准。

6. Gas机制:
- Base Fee + Priority Fee
- Gas Used × Effective Gas Price = Total Fee
- 根据场景优化:能等就等,紧急就加Priority Fee
- 使用Layer 2降低成本

7. 失败定位:
- Out of Gas: Gas Limit不够
- Revert: 合约拒绝,查看错误信息(余额不足、未授权、逻辑不满足)
- 排错清单:Status → Gas → Logs → Internal Txns → Input Data

8. 确认与最终性:
- 确认数越多越安全,但也越慢
- 小额1-2确认,大额等最终性
- 了解不同链的确认机制

9. 可疑交易识别:
- 未预期的Approval或Transfer
- 未验证的合约
- 异常的Gas消耗
- Internal Txns中的意外转账
- 使用安全工具辅助判断

实践能力:

通过今天的学习,你应该能够:
- 独立查询和验证任何交易
- 读懂交易详情页的所有字段
- 定位失败交易的准确原因
- 估算合理的Gas费用
- 识别可疑和恶意的交易
- 使用多种工具辅助分析

安全习惯:

1. 每次重要操作后,去区块浏览器验证:
- Status是否Success?
- Logs是否符合预期?
- Internal Txns有无异常?

2. 定期(如每月)审查授权:
- 用Revoke.cash查看授权列表
- 撤销不再使用或不认识的授权

3. 签名前思考:
- 看清Input Data,理解你在调用什么
- 检查合约是否已验证
- 评估Gas成本是否合理

4. 遇到失败,冷静排错:
- 按清单逐步检查
- 参考社区讨论
- 小额测试验证方案

下节课预告:

下次课,我们将学习:
- 智能合约的深度分析:如何读懂合约代码
- 常见的合约模式:ERC-20、ERC-721、Proxy、Ownable

这些都是进阶技能,学会后你就能独立审查任何项目,做出明智的决策。

最后,送你一句话:在Web3中,验证比信任更重要。不要盲信任何人的说法,自己去链上查,自己去验证,你就永远不会被欺骗。
